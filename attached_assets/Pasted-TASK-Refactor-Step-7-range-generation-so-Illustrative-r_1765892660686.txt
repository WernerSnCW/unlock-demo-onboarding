1) What must change (behaviour)

Neutral must render approximately: 40.8–44.8 (rounded as UI needs).

Range width must be ±10 percentage points from current (before clamping).

Ranges must be visibly asymmetric when beliefs are non-neutral.

Example requirement (equity current 42.8% with pro-risk / growth beliefs):
Preference-leaning must produce something like: 39–53 (asymmetric), not 33–53 or 39–49 unless constrained.

2) New computation model (must implement)

For each displayed allocation row x (e.g., equity, bond, cash, property; global, uk; wrappers):

2.1 Compute scenario movement cap

capNeutral = 2

capGuardrail = 3

capPreference = 10

2.2 Compute belief “pressure” per row in [-1, +1]

Create a deterministic mapping from Step 6 axis scores to allocation-row pressures:

equity_pressure combines: smallCap(+), tech(+), value(+ small), quality(+ small), volatilityAversion(−)

bond_pressure combines: volatilityAversion(+), inflationHedge(+ if towards / − if away)

cash_pressure combines: volatilityAversion(+), plus liquidity guardrail needs if liquidity is amber

property_pressure combines: property/alternatives signals from intake/persona traits (+), and must be reduced to 0 if illiquids is amber/red

regional pressures:

uk_pressure uses UK bias axis only

global_pressure should move inversely to uk_pressure (so totals remain coherent)

Make sure the “pressure” values are clamped to [-1, +1] and are reproducible for the same inputs.

2.3 Apply scenario-specific pressure scaling

Neutral baseline: pressureScale = 0

Guardrail-first: pressureScale = 0.25

Preference-leaning: pressureScale = 1.0

2.4 Convert pressure into skew (in percentage points)

skew = pressure * skewStrength

Use: skewStrength = cap * 0.6

(So neutral skew = 0; guardrail skew small; preference skew noticeable)

2.5 Build an asymmetric range around current using skew

Instead of symmetric [current-cap/current+cap], implement:

min = current - cap + min(0, skew)

max = current + cap + max(0, skew)

This forces asymmetry:

positive skew extends the upper bound more than the lower bound

negative skew extends the lower bound more than the upper bound

2.6 Apply hard clamps/budgets AFTER range creation

Enforce budgets from Step 5:

If concentration is AMBER/RED:

do not allow equity max to increase beyond current by more than a conservative amount (e.g. +0 to +2) unless you can prove concentration won’t worsen

If liquidity is AMBER/RED:

do not allow cash min to fall below current (or below a runway-preserving threshold)

If illiquids is AMBER/RED:

do not allow property/illiquid max to exceed current

Also ensure totals remain plausible:

After generating per-row ranges, normalise so that:

lower bounds don’t sum above 100%

upper bounds don’t sum below 100%
If needed, proportionally tighten non-binding sleeves to restore coherence (without changing the directionality).
